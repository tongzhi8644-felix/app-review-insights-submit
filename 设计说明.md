# App Review Insights — 设计说明

本文档对应本仓库**实际实现**（Flask Web + 分析流水线 + 模型语义层），不是抽象方案。

---

## 一、项目是什么

本系统是一个**本地可运行的 Web 分析工具**：用户输入美国区 App Store 链接和分析目标（可选上传 JSON/CSV），点击 Start 后自动跑完整流水线，并在页面展示进度与全部交付物。

主演示 App 默认链接为考核示例：

`https://apps.apple.com/us/app/workout-for-women-home-gym/id839285684`

但分类、发现、PRD、用例均由**当次评论 + 目标**动态生成，**没有**写死该 App 的问题库。

---

## 二、设计思路

### 1. 为何做成「流水线」而不是单次脚本

考核要求覆盖：划定范围 → 采集 → 清洗 → 动态分析 → 证据评估 → PRD/版本 → 测试用例 → 追溯校验 → UI 进度与交付物。  
因此以一次 `AnalysisRun` 为中心，由 `app/services/workflow.py` 按固定阶段推进，每阶段写 `StageLog`，最终把结果写入 `result_json` 并落库。

实际阶段（与代码 `STAGES` 一致）：

1. `scope` — 记录 App ID、美国区、分析目标  
2. `collect` — RSS / 导入 / 样本 / 标注缓存  
3. `clean` — 规范化、去重、按目标软过滤  
4. `classify_analyze` — **DeepSeek** 动态主题 + findings  
5. `plan_prd` — **DeepSeek** 版本计划 + PRD + 需求  
6. `test_cases` — **DeepSeek** 测试用例  
7. `validate` — 确定性追溯校验  
8. `persist_finalize` — 汇总交付物，进度 100%

### 2. 规则与模型怎么分工（本项目的取舍）

| 环节 | 实现位置 | 方法 | 原因 |
|------|----------|------|------|
| URL 解析、RSS 拉取 | `collector.py` | 规则 + HTTP | 可复现；公开接口 |
| 清洗、哈希去重、空评丢弃 | `cleaner.py` | 规则（SHA-256） | 答案唯一 |
| 目标软过滤（低分/高分/版本） | `cleaner.filter_by_goal` | 轻量规则 | 只缩小候选，不做语义定性 |
| 主题/分类/发现 | `analyzer.py` + `llm.py` | **DeepSeek** | 动态发现，非固定词表 |
| PRD / 版本 / 需求 | `planner.py` | **DeepSeek** | 需归纳与优先级 |
| 测试用例 | `testgen.py` | **DeepSeek** | 需挂需求与源评论 |
| 追溯校验 | `validator.py` | 规则 | 非法 ID 必须可证明地剔除 |

原则：**能确定的不用模型；必须语义泛化的才调 DeepSeek。**

### 3. 数据从哪来（本项目真实做法）

**在线主路径**（非 HTML 爬页面）：

```
https://itunes.apple.com/us/rss/customerreviews/page={n}/id={app_id}/sortby=mostrecent/json
```

- 国家固定 `us`（考核要求评论来自美国区）  
- 页数由 `.env` 的 `MAX_REVIEW_PAGES` 控制（默认 10）  
- 页间 `sleep 0.35s`，避免对目标站异常压力  
- 代码：`app/services/collector.py` → `fetch_reviews_rss`

**其它输入（同一套后续流水线）：**

| 方式 | 触发 | data_source 标记 |
|------|------|------------------|
| 上传 JSON/CSV | 表单 `file` | `import_json_csv` |
| 勾选 sample | `use_sample` | `sample_file_labeled` |
| RSS 失败且允许兜底 | `ALLOW_CACHED_FALLBACK=1` | `cached_rss_labeled` |

样本与缓存目录：`data/sample_reviews.json`、`data/sample_reviews.csv`、`data/cached_analysis/`（文件内有 `_label` 标注）。  
**标注缓存只用于离线审阅/失败兜底，不能替代配置好网络与模型后处理新输入。**

### 4. 为何要「抽样再进模型」

实测一次 RSS 可拉约 250 条评论。若全文塞进一次调用，易导致 JSON 截断、findings 为空。  
因此 `analyzer._select_review_sample` **分层抽样最多 40 条**（优先低分 → 中分 → 高分），标题/正文截断后再调模型；空结果会重试一次，仍空则用**标注过的**启发式填充（日志写明），避免 UI 空白。

## 三、实现过程（对应仓库结构）

### 1. 技术组成（本仓库）

| 部分 | 具体选择 |
|------|----------|
| Web | Flask 3.1 + Jinja 模板 + 原生 JS（`app/static/js/app.js`） |
| 持久化 | SQLAlchemy；主库 MySQL（`PyMySQL`）；可选 `USE_SQLITE=1` 本地冒烟 |
| 表结构 | `app/models.py` + `sql/schema.sql`（`analysis_runs`、`stage_logs`、`reviews_raw/clean`、`classifications`、`findings`、`prd_documents`、`test_cases`、`validation_results`） |
| HTTP 采集 | `requests` |
| 模型客户端 | `openai` SDK，OpenAI 兼容协议（`app/services/llm.py`） |
| 启动 | `python run.py` → `create_app()` → `http://127.0.0.1:5000` |

依赖见 `requirements.txt`；部署见 `deploy.txt`。

### 2. 后端接口（实际路由）

| 方法 | 路径 | 作用 |
|------|------|------|
| GET | `/` | 分析首页 |
| GET | `/runs/<id>` | 结果页（可刷新） |
| POST | `/api/analyze` | 启动一次分析（表单/JSON） |
| GET | `/api/runs/<id>` | 查询进度与 `result` |
| GET | `/api/runs/<id>/reviews/raw` | 原始评论 |
| GET | `/api/runs/<id>/reviews/clean` | 清洗后评论 |
| GET | `/api/meta/methods` | 方法与数据源说明 |

前端点 Start → `POST /api/analyze` → 轮询 `/api/runs/<id>` 更新进度条与阶段列表。

### 3. 前端交付物 Tab（对应考核第 10 条）

页面 `Deliverables` 分区：

| Tab | 内容 |
|-----|------|
| Findings | 发现、置信度、源 review_id、摘录、假设标记 |
| PRD | 版本计划、需求、PRD markdown |
| Test cases | 步骤、期望、关联需求与评论 |
| Classification | 每条评论的 topics / sentiment |
| Reviews | Raw + Cleaned 两张表 |
| Validation | 校验摘要、issues、revisions |
| Raw JSON | 整次 run 序列化结果 |

### 4. 实现迭代顺序（与模块落地一致）

1. 配置与模型：`config.py`、`models.py`、`extensions.py`  
2. 采集清洗：`collector` → `cleaner`  
3. 模型链路：`llm` → `analyzer` → `planner` → `testgen` → `validator`  
4. 编排：`workflow.start_run` / `_execute`  
5. API + UI：`routes` + `templates` + `static`  
6. 可演示材料：`data/` 样本、缓存、示例输出；文档 `README` / `deploy.txt` / 本文

## 四、AI 的使用方式（本项目）

### 1. 使用的模型

运行时语义分析使用 **DeepSeek**（模型名 `deepseek-chat`，经 OpenAI 兼容 Chat Completions 接入，例如 Silra 网关）。

相关环境变量（见 `.env.example`，密钥不进仓库）：

- `AI_ENABLED=1`
- `OPENAI_API_KEY`
- `OPENAI_BASE_URL`（支持填 `/v1` 或完整 `.../chat/completions`，代码会归一化）
- `OPENAI_MODEL=deepseek-chat`
- `OPENAI_TEMPERATURE`（默认 0.7）
- `OPENAI_MAX_TOKENS`（默认可调高，避免长 JSON 截断）

封装：`app/services/llm.py` 的 `chat_json()`：优先 `response_format=json_object`，网关不支持则降级普通补全，再解析 JSON（含去掉 markdown 代码围栏）。

> 开发时可用编码助手加速写代码；**考核认定的 AI 能力是运行时这条模型链路**，由 `workflow` 在 `classify_analyze` / `plan_prd` / `test_cases` 阶段真实调用。

### 2. DeepSeek 在流水线中的三次调用

| 阶段 | 函数 | 输入 | 输出 |
|------|------|------|------|
| classify_analyze | `analyzer.analyze_reviews` | 抽样评论 + 分析目标 + app_id | `classifications` + `findings`（含 source_review_ids、confidence、excerpts、is_assumption 等） |
| plan_prd | `planner.build_prd` | findings + 清洗统计 | `version_plan` + `requirements` + `prd_markdown` |
| test_cases | `testgen.generate_test_cases` | requirements | `test_cases`（挂 `requirement_id` 与 `source_review_ids`） |

Prompt 共性约束（见各模块 `SYSTEM` 字符串）：

- 面向**任意 App**，禁止固定问题 taxonomy  
- **禁止编造** review_id  
- 只返回 JSON  
- 弱证据降置信度或标 `is_assumption`

### 3. 失败与兜底（代码真实行为）

1. 未配置 Key / 调用失败 → 阶段日志 `warning`  
2. `ALLOW_CACHED_FALLBACK=1` 时可读 `data/cached_analysis/{app_id}_bundle.json`（标注缓存）  
3. 否则走 `heuristic_analyze` / `heuristic_prd` / `heuristic_test_cases`（明确 `method: heuristic_fallback_labeled`）  
4. 模型返回空 findings → 用同批真实 review_id 做启发式填充，并写日志，避免校验后整页「No findings」误导  

**不伪造评论原文**；数据不足时在 `data_limitations` / Validation 中说明。

### 4. 防幻觉与「有证据才保留」

后置闸门：`validator.validate_traceability`

- findings / requirements / test_cases 中的 review_id 必须落在当次清洗范围内  
- 无效 ID 删除；完全无证据的发现/需求移除或标假设  
- 用例孤儿需求检测；校验结果写入 `validation_results` 并在 UI **Validation** Tab 展示  

因此 UI 上能区分：**模型生成的结论** vs **规则校验后的可追溯结论**。

## 五、如何演示（对应本系统 UI）

1. `python run.py`，打开 `http://127.0.0.1:5000`  
2. 填写目标（如 subscription / low-rating），可不勾 sample，点 Start（在线：RSS + DeepSeek）  
3. 看进度到 100%，且 `classify_analyze` 为模型成功（非 cache/heuristic 警告）  
4. 依次打开 Findings → PRD → Test cases → Classification → Reviews → Validation  

导入能力：上传 `data/sample_reviews.json` 或 `.csv`。  
离线审阅：勾选 sample 或使用 `data/cached_analysis/`、`data/sample_output/`。

---

## 六、一句话总结（可口述）

> 本项目用 Flask 把「RSS/导入 → 规则清洗 → DeepSeek 语义产出 → 规则追溯校验」串成可观测流水线，所有交付物在同一 UI 里展示；模型只做动态语义，证据是否成立由校验层用真实 review_id 把关。
